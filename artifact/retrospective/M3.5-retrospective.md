## 🧭 方案合理性复盘

### 做对的几件事

1. **thumb 算法版本号 + dimsVerified 版本号**：两级版本号让算法升级时老数据自愈，不需要用户"手动重建"；`normalizeDimsVersion(true)=1` 让 boolean 老数据平滑迁移
2. **`grain://thumb?v=N` cache-bust**：没改协议本身，只加查询参数，WebContents 视为新 URL 必然重 fetch —— 最小侵入
3. **`photo:repaired` push 事件**：主进程 → 渲染进程的单向通知，UI 订阅即自动刷新，比轮询优雅
4. **专家级滑块优化的三层协同**：输入层 rAF + 渲染层 rAF + 算法层 curve —— 每层解决一类问题，叠加起来用户体验质变
5. **`removePhotoRecords` 的安全契约**：`path.resolve` + 目录前缀校验 + `fs.unlinkSync` spy 测试三重防护，**硬盘原图绝对不动** —— 用测试把契约永久锁定

### 踩过的坑

1. **对"视觉 bug"的经验主义**：第一次以为算法问题反复改 `sharp.rotate`，浪费 1 轮迭代；应该**先查数据真值 + 磁盘像素**再改代码（下轮执行 4 步 SOP）
2. **`preview-cache` 写了没清**：为了修 data URL 过大 fetch 失败临时引入文件路径方案，但没有同步的 GC 策略；好在阈值 2MB 下实际很少命中（日常只写几次），但属于"埋了就忘"的风险
3. **CPU 镜像没同步改**：改 GPU shader 时 snapshot 未挂，**误以为**是好事（无回归）；实际是 CPU 镜像独立不联动，**防退化强度本就有限**。这个发现比改动本身更重要

### 下轮建议

1. **先补上 CPU 镜像 curve() 镜像 + HSL 完整版**（快速清还本轮引入的债）
2. **`preview-cache` GC**：启动时扫 + LRU 上限（复用 RAW 缓存的 `diskLRU.ts` 代码）
3. **执行全项目复盘报告 §6.2 的 M4 修订方案**：编辑器 UI 闭环（撤销/重做/前后对比/保存为滤镜）+ 顺手加一个 "GPU benchmark in-page" 原型
4. **AGENTS.md 加第 7 条**：视觉 Bug 诊断 SOP 变成强制检查项，永久免疫本轮两次误诊
