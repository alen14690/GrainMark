# E2E 测试体系批判性复盘 · 2026-05-01

> Pass T1–T4 交付后,用户要求"不要表面绿、实际无法发现问题的测试方案"。本次复盘通过
> 3 次蓝军 mutation 实证 + 源码审计,暴露 6 个严重问题。本文件为修复依据,
> 修复完成后将作为档案保留,记录问题 → 根因 → 修复手段 → 真红验证。

## 问题清单

### 1. 【致命】整套 E2E 对"滤镜 pipeline 真生效"零防护

**现象**:把 `useEditStore.loadFromPreset(preset)` 内部改成 `s.currentPipeline = null`(mutation),
`editorJourney.spec.ts` 的 6 条用例 **6/6 全绿**。

**根因**:
- J3 只断言 DOM class `brand-violet` + 顶栏 sub 文案——这都由 `useAppStore.activeFilter`
  驱动,**与滤镜管线是否真的加载到 editStore 无关**
- J4/J5 只看 Slider 的 `aria-valuenow` 和 EDITED badge——这些是 editStore 的 `_dirty` 标记,
  `currentPipeline=null` 时照样能 `setTone` → ensurePipe 会凭空新建 pipeline
- J6 只看 EDITED 消失——`loadFromPreset(null)` 把 `_dirty` 置 false 即满足,
  **完全不检查 currentPipeline 的实际结构**

**后果**:滤镜管线核心契约——"点击 filterRow 后 editStore.currentPipeline 应等于
preset.pipeline 深拷贝"——没有任何一条 E2E 断言覆盖。用户反映"切换滤镜没效果"这类 bug,
现有测试不会红。

**修复**:在 J3 追加断言,通过 `__grainEditStore` 读 `getState()`,验证:
- `currentPipeline` 不为 null
- `baselineFilterId === 'kodak-portra-400'`
- `currentPipeline` 至少含 tone/colorGrading/lut3D 任一字段(Portra 400 preset 含这些)

由于 `__grainEditStore` 挂载条件是 `import.meta.env.DEV || MODE === 'test'`,
production build(E2E 实际跑的构建)并不注入。需改用 preload 暴露的 `grain.testMode`
读 `process.env.GRAINMARK_TEST === '1'`,仅测试环境挂载 store,生产仍然不暴露。

### 2. 【严重】packaged P3 没有真正过 native binding

**现象**:P3 只调 `photo:list`(返回空数组)。`listPhotos` 内部只读
`JsonTable photos.json`,**完全不触碰 sharp / exiftool-vendored / better-sqlite3**
任意一个 native binding。asar unpack 有漏项也发现不了。

**修复**:将 P3 改为"注入一张真实 fixture 走 photo:import":
- `photo:import` 内部会调 sharp(生成 thumb/preview)+ exiftool(读 EXIF)——**两者都是 native binding**
- 断言返回的 Photo 至少含 `id`/`name`/`width`/`height`(后两者来自 sharp)
- 断言 thumb 文件真实存在磁盘(证明 sharp 成功编码 + 主进程有写权限)

### 3. 【细节】launchApp 里的 `--high-dpi-support=1` 是无效 flag

**现象**:Chromium 只认 `--force-device-scale-factor`,不存在 `--high-dpi-support`
(Electron 命令行不支持)。

**修复**:删除该 flag,保留注释说明只用 `--force-device-scale-factor=1` 即可。

### 4. 【代码洁癖】`selectors.ts` 自己违反自己的规则

**现象**:模块注释声明"spec 文件不允许直接写 `page.getByTestId`",
但 `editorJourney.spec.ts` L90/L99/L146/L150/L157 多处直接 `page.getByTestId('editor-tab-*')`。
同时 `editorRedoBtn`/`editorExportBtn` 两个 export 没有任何 spec 引用。

**修复**:
- spec 里 tab 切换改用 selectors 的 `editorTabFilters`/`editorTabAdjust`
- 删除从未被引用的 `editorRedoBtn` / `editorExportBtn`

### 5. 【冗余】smoke.spec.ts 5 条里有冗余/重复

**现象**:
- 第 1 条 "window.grain 可用"——launchApp 里的 `waitForFunction` 已隐式保证,这里再断言重复
- 第 4/5 条"点击 Editor / 回到 Library"——editorJourney J1/J2 会更强力地覆盖导航

**修复**:smoke 只保留 2 条——(a) sidebar 渲染 + 4 个主导航可见; (b) 空态
import 按钮可见且 photo-grid 不存在。其余交由 editorJourney 覆盖。

### 6. 【稳定性】J5 Undo 循环潜在 flaky

**现象**:`for (let i = 0; i < 40; i++) { if (!await undo.isEnabled()) break; await undo.click() }`
每次 click 都重新等 actionable,40 轮最坏可能接近 60s 总超时。实测一次 ~55s。

**修复**:
- 把"曝光值"作为终止条件——只要 aria-valuenow 回到 ±0.05 即退出循环,避免无意义继续点
- 每次 click 加 `{ timeout: 1500 }`,超时更快暴露真卡死
- 最大轮次从 40 降到 25(实际 history 容量远小于此)
