## 🔍 方案合理性复盘

### 👍 好的决策

1. **不做 demosaic**：前期花 1 整天做了 3 个 LibRaw 候选包的技术 spike，全部阻塞（webworker 不兼容 / 无 Windows 预构建 / 2-12 stars）；抽嵌入 JPEG 是**正确的 90 分方案**
2. **sha1(path + mtime + size) 作为 cache key**：任何源文件改动自动失效，不需要手动 bust
3. **懒修复 + 每次最多 8 条**：避免首次进入图库时的一次性吞吐，UX 响应立刻可用
4. **orientation 修复两头拦截**：RAW 原文件 EXIF 为准 + 显式 rotate(angle) → 不信任内嵌 JPEG 的 EXIF（Sony ARW 常常谎报 orientation=1）
5. **UnsupportedRawError 带判别 code**：便于未来针对不同错因给不同 UX 文案
6. **resolvePreviewBuffer 作为唯一入口**：thumbnail / preview / grain:// 三处消费者共享同一修复点

### 🤔 可能改进

1. **sharp 的 rotate(angle).withMetadata 之前那版冗余**：说明我对 sharp 默认输出行为有误解，review 时补测试可以更快发现
2. **磁盘 LRU 的 evict 到 80% 水位线是硬编码**：未来 M10 Settings 应暴露「清理到 X%」选项
3. **cache 大小 64MB..64GB 夹紧**：用户反馈前不改
4. **非 RAW 也走 resolvePreviewBuffer → fs.readFile Buffer → sharp**：比之前的 sharp(filePath) 多一次内存拷贝；24MP JPEG ≈ 4MB 可忽略，但如果后续接 100MP 医学图像要重测
